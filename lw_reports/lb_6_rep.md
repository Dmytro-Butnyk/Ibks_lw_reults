# Лабораторна робота № 6

**Тема: Windows Crypto API NG**

**Мета:** освоїти на практиці використання Windows Crypto API NG в задачах шифрування, хешування і електронного цифрового підпису.

## Порядок виконання

1. **Проектування додатка.** Розроблено консольний додаток `cryptong.exe`, який приймає параметри з командного рядка: `<hash|test|enc|dec> <input file> [<signature>] <key>`.
2. **Робота з файлами.** Реалізовано завантаження вхідних файлів з використанням технології відображення на пам'ять (Memory-Mapped Files) для оптимізації швидкості роботи.
3. **Хешування.** Реалізовано обчислення хешу алгоритмом SHA-256. Життєвий цикл хешу виконано за стандартом: `BCryptOpenAlgorithmProvider` → `BCryptCreateHash` → `BCryptHashData` → `BCryptFinishHash` → `BCryptDestroyHash` → `BCryptCloseAlgorithmProvider`.
4. **Генерація ключа.** Для операцій шифрування/розшифрування реалізовано механізм перетворення текстового пароля користувача у двійковий ключ для AES за допомогою хешування SHA-256.
5. **Шифрування файлу.** Використано алгоритм AES-256. Дані успішно зашифровано та розшифровано.
6. **ЕЦП.** Реалізовано базову підготовку до підписання файлів з використанням RSA-2048.

## Висновок
Під час виконання лабораторної роботи було освоєно сучасну криптографічну підсистему Windows — Crypto API NG (CNG). Було на практиці реалізовано роботу з провайдерами алгоритмів, створення та управління криптографічними ключами, хешування (SHA-256) та симетричне шифрування (AES-256). Завдяки використанню відображення файлів на пам'ять, додаток працює ефективно навіть з великими обсягами даних.

## Відповіді на контрольні запитання

**1. Що таке "Ephemeral Key" (ефемерний ключ) і чим він відрізняється від персистентного ключа?**
Ефемерний ключ існує лише в оперативній пам'яті протягом часу життя його об'єкта (handle). Після закриття він безповоротно знищується. Персистентний ключ зберігається на постійних носіях (диск, SmartCard) через Key Storage Provider (KSP) та має власне ім'я для повторного використання.

**2. Яку роль відіграє процес LSA (Local Security Authority) в ізоляції ключів у системі?**
Процес LSA (`lsass.exe`) забезпечує ізоляцію та захист постійних ключів (KSP). Ключі не завантажуються в адресний простір програми користувача; натомість програма спілкується з LSA через RPC для виконання криптографічних операцій, що унеможливлює викрадення ключа безпосередньо з пам'яті програми.

**3. Поясніть механізм "Key Derivation" (KDF). Які алгоритми KDF підтримує Windows CNG?**
Key Derivation Function (KDF) — це процес створення одного або кількох криптографічних ключів із базового секрету (наприклад, пароля) за допомогою псевдовипадкових функцій або хешування. CNG підтримує алгоритми KDF на базі SP800-108, SP800-56A, PBKDF2, а також специфічні для протоколів (TLS PRF, IKEv2).

**4. Що таке "Провайдер" (Provider) у контексті CNG і як працює механізм пріоритетності провайдерів?**
Провайдер — це DLL-модуль, який реалізує набір криптографічних примітивів (хеші, шифри). У CNG можна передати `NULL` замість імені провайдера, і система сама обере найкращий доступний провайдер на основі пріоритетів, налаштованих у системному реєстрі (наприклад, апаратний HSM-провайдер отримає пріоритет над програмним).

**5. Яким чином програміст може примусово вибрати конкретний провайдер (наприклад, MS_PRIMITIVE_PROVIDER) замість використання замовчуваного?**
Програміст має передати точну назву провайдера (наприклад, `MS_PRIMITIVE_PROVIDER`) у якості другого аргументу `pszProvider` до функції `BCryptOpenAlgorithmProvider` або `NCryptOpenStorageProvider` замість `NULL`.
