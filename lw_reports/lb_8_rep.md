# Лабораторна робота № 8

**Тема: Microsoft Authenticode**

**Мета:** освоїти на практиці використання технології цифрового підпису коду як методу попередження розповсюдження зловмисного програмного забезпечення.

## Порядок виконання

1. **Реалізація перевірки підпису.** Розроблено базову функцію перевірки цифрового підпису `VerifyFileSignature` з використанням Windows API функції `WinVerifyTrust`. Вона перевіряє PKCS#7 підпис, сертифікат та ланцюжок довіри.
2. **Сканування активних процесів.** Консольний додаток за допомогою `CreateToolhelp32Snapshot` перебирає всі запущені процеси в системі. Кожен файл образу процесу передається до функції `VerifyFileSignature`. Процеси з відсутнім або недійсним підписом виводяться на екран.
3. **Розробка DLL-перехоплювача.** Створено динамічну бібліотеку `appcert.dll` з експортованою функцією `CreateProcessNotify`. Під час запуску нових процесів ця функція отримує керування, аналізує стан `uReason` (наприклад, `APPCERT_IMAGE_OK_TO_RUN`), перевіряє підпис образу і, в разі його відсутності, передає повідомлення головному додатку через Named Pipe.
4. **Реєстрація та моніторинг.** Консольний додаток автоматично реєструє абсолютний шлях до розробленої DLL у гілці реєстру `HKLM\SYSTEM\CurrentControlSet\Control\Session Manager\AppCertDlls`. Додаток переходить в режим сервера іменованого каналу і виводить в режимі реального часу повідомлення від DLL. При завершенні роботи (`Ctrl+C`) реєстр автоматично очищується для уникнення збоїв операційної системи.

## Висновок
Під час виконання лабораторної роботи було вивчено роботу технології Microsoft Authenticode. Практично доведено, що перевірка цифрового підпису є ефективним методом виявлення нелегітимного ПЗ. Використання механізму `AppCertDlls` дозволяє інтегрувати власні засоби перевірки цілісності коду безпосередньо в процес створення нових процесів ядром ОС, що є потужним інструментом для розробки засобів класу Endpoint Detection and Response (EDR).

## Відповіді на контрольні запитання

**1. Яке основне призначення технології Мicrosoft Authenticode?**
Основне призначення — забезпечення безпеки програмного забезпечення у Windows шляхом підтвердження справжності видавця та цілісності коду (гарантія того, що код не був змінений або заражений вірусами після підписання).

**2. Де зберігається підпис Authenticode у РЕ-файлі (Portable Executable)?**
Підпис зберігається всередині самого PE-файлу в спеціальному розділі, який називається Attribute Certificate Table (його адреса та розмір вказуються в Security Directory).

**3. Які типи файлів підтримують підпис Authenticode?**
Підпис підтримують будь-які виконувані файли формату PE (`.exe`, `.dll`, `.sys`, `.ocx`), а також кабінетні архіви (`.cab`), скрипти PowerShell (`.ps1`) та інсталятори (`.msi`).

**4. Чи входить Authenticode-підпис у хеш РЕ-файлу?**
Ні. Перед обчисленням хешу алгоритм тимчасово виключає з розрахунку поле Checksum в заголовку PE та сам розділ Security Directory (де лежить підпис). Якби підпис входив у хеш, його додавання до файлу змінило б сам хеш, що зламало б весь механізм.

**5. Що таке SPC (Software Publisher Certificate)?**
Це сертифікат видавця програмного забезпечення стандарту X.509 (формат PKCS#7), який видається довіреним центром сертифікації (CA) розробнику для створення підпису Authenticode.

**6. Яка опція лінкера в Visual Studio готує файл до підпису, резервуючи місце для Certificate Table?**
Visual Studio дозволяє встановити прапорець `/INTEGRITYCHECK` у параметрах лінкера, що встановлює біт `IMAGE_DLLCHARACTERISTICS_FORCE_INTEGRITY`. Однак саме місце під сертифікат зазвичай не резервується заздалегідь — утиліта `signtool.exe` просто дописує таблицю в кінець файлу та оновлює заголовки.

**7. Яка різниця між «Signed» i «Trusted»?**
"Signed" (підписаний) означає, що файл містить криптографічно правильний підпис і його хеш збігається. "Trusted" (довірений) означає, що сертифікат, яким зроблено підпис, належить ланцюжку довіри (Chains of Trust), корінь якого встановлено в сховищі довірених кореневих сертифікатів (Root CA) операційної системи.

**8. Навіщо потрібна позначка часу (timestamp) y Authenticode?**
Позначка часу (яку видає незалежний Time Stamping Authority сервер) криптографічно доводить точний час підписання файлу. Це гарантує, що підпис залишатиметься дійсним навіть після того, як термін дії самого сертифіката видавця закінчиться.

**9. Яка роль кореневого сертифіката (Root CA)?**
Кореневий сертифікат є кінцевою точкою довіри в системі (Trust Anchor). Якщо ланцюжок сертифікатів, яким підписано файл, не зводиться до відомого системі Root CA, система не довірятиме цьому файлу, навіть якщо сам підпис математично коректний.

**10. Чи можна підписати вже підписаний файл ще раз?**
Так, можна. Це називається подвійним підписом (Dual-signing). Найчастіше це використовувалося під час переходу з SHA-1 на SHA-256, щоб файл містив обидва підписи для сумісності зі старими та новими ОС.

**11. Якщо сертифікат для підпису коду закінчився, але є дійсна позначка часу, чи вважається підпис дійсним?**
Так. Якщо файл був підписаний і отримав позначку часу в момент, коли сертифікат ще був дійсним, підпис вважається довіреним безстроково (доки алгоритм хешування не буде скомпрометовано).
