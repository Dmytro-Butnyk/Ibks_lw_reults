# Лабораторна робота № 7

**Тема: Метод розподілу ключів Діффі-Хеллмана**

**Мета:** освоїти на практиці використання методу розподілу ключів Діффі-Хеллмана в задачах передавання повідомлень по незахищеним каналам.

## Порядок виконання

1. **Проектування архітектури.** Розроблено два консольних додатки (клієнт та сервер) на мові C++ із використанням Winsock API (режим TCP, SOCK_STREAM) для передачі даних по мережі.
2. **Мережева взаємодія.** Серверний додаток реалізовано як багатопотоковий (`std::thread`), що дозволяє одночасно обслуговувати кілька клієнтських підключень на порту 54000.
3. **Генерація та обмін ключами.** Використано Windows Crypto API NG (CNG). Клієнт та сервер генерують власні пари ключів за алгоритмом ECDH (на еліптичній кривій P-256). Відкриті ключі експортуються (`BCryptExportKey`) та передаються через сокет.
4. **Отримання симетричного ключа.** Після отримання відкритих ключів парнтера, кожна сторона генерує спільний секрет (`BCryptSecretAgreement`). За допомогою KDF-функції (`BCryptDeriveKey`) на основі алгоритму SHA-256 обчислюється 256-бітний ключ для AES.
5. **Шифрування та передача файлу.** Клієнтський додаток шифрує вибраний файл алгоритмом AES-256 (режим CBC). Для правильної обробки серверу передається структурований заголовок із магічним числом, розміром шифротексту та вектором ініціалізації (IV). Сервер приймає файл, розшифровує його та зберігає на диск.

## Висновок
Під час лабораторної роботи було успішно застосовано протокол Діффі-Хеллмана на еліптичних кривих (ECDH) для обміну ключами через незахищений канал (TCP). Практично доведено, що комбінація асиметричної криптографії для обміну сесійними ключами та симетричної криптографії (AES-256) для передачі об'ємних даних є ефективним та безпечним архітектурним рішенням.

## Відповіді на контрольні запитання

**1. Яка основна мета методу Діффі-Хеллмана (DF)?**
Основна мета — відкритий розподіл криптографічних ключів по незахищених каналах зв'язку, що дозволяє двом сторонам отримати спільний секретний ключ без його попереднього передавання по мережі.

**2. На обчислювальній складності якої задачі базується безпека класичного алгоритму DF?**
Безпека алгоритму базується на складності обчислення дискретних логарифмів у скінченному полі.

**3. Які параметри є відкритими (public) у протоколі DH?**
Відкритими параметрами є великі прості числа $q$ (генератор) і $n$ (модуль), а також відкриті ключі користувачів $K_A$ та $K_B$, які передаються по незахищеному каналу.

**4. Які розміри в бітах простих чисел вважається мінімально безпечним сьогодні для класичного DH згідно з NIST?**
Згідно з актуальними рекомендаціями NIST (SP 800-57), мінімальним безпечним розміром модуля для класичного DH є 2048 біт (забезпечує 112 біт симетричної стійкості).

**5. Чому сесійний ключ DF не використовують напряму як AES-ключ?**
Отриманий сесійний секрет може мати ентропію, розподілену нерівномірно, або не відповідати необхідній довжині ключа для вибраного алгоритму (наприклад, рівно 256 біт для AES). Тому на його основі генерується ключ за допомогою KDF (Key Derivation Function) або хешування.

**6. Що таке досконала пряма секретність (англ. Perfect Forward Secrecy, PFS) у контексті DF?**
PFS — це властивість криптографічного протоколу, яка гарантує, що компрометація довготривалих (статичних) ключів користувачів у майбутньому не призведе до розшифрування перехоплених раніше сесійних даних.

**7. Яка властивість робить DH корисним для PFS?**
DH підтримує генерацію ефемерних (одноразових) пар ключів (Ephemeral Diffie-Hellman, EDH/DHE). Оскільки для кожної нової сесії генеруються нові випадкові секрети, дізнатись минулий сесійний ключ після його знищення неможливо.

**8. Яка перевага алгоритму Діффі-Хеллмана на основі еліптичних кривих (ECDH) перед класичним DH?**
ECDH забезпечує аналогічний або вищий рівень криптостійкості при значно менших розмірах ключів. Наприклад, 256-бітний ключ ECDH забезпечує такий самий захист, як ~3072-бітний ключ класичного DH чи RSA. Це зменшує навантаження на процесор і мережу.

**9. Що таке KDF (Key Derivation Function) у контексті обміну ключами DH/ECDH?**
KDF — це функція, яка використовує спільний секрет, отриманий після алгоритму DH/ECDH, для криптографічно безпечного "виведення" (генерації) одного або декількох симетричних ключів необхідної довжини (наприклад, для AES).

**10. Чи можна використовувати один і той самий довготривалий (static) ключ ECDH для багатьох сесій?**
Технічно так, це називається Static ECDH. Однак це вкрай не рекомендується, оскільки при цьому втрачається властивість Perfect Forward Secrecy (PFS). Якщо цей статичний ключ буде скомпрометовано, зловмисник зможе розшифрувати весь перехоплений трафік усіх попередніх сесій.
